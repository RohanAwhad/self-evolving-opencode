# Brainstorm: Self-Evolving Skills Pipeline

> **Note**: This document captures the design discussions. Final design is in `specs/008-013`. Some details evolved during spec writing (e.g., two-queue design, single `--evolve` mode, synthesized vs curator responsibilities). See specs for authoritative design.

## Original Pipeline (built)

```
OpenCode SQLite DB
  → extract goals (LLM tool-use)
  → check if achieved (LLM verdict)
  → summarize conversation threads (Goal/Intent/Outcome/Tool Usage)
  → cluster goals by semantic similarity (embeddings + agglomerative)
```

## New Pipeline (building)

Two modes: **initial run** (batch over all past conversations) and **periodic runs** (daily, new conversations only).

### Initial Run

No existing rules — reflector runs in insight-only mode, curator builds full skills from scratch.

```
All sessions
  → extract all goals
  → cluster (embeddings + agglomerative)
  → sort clusters by size DESC
  → for each cluster (above min size):
      1. Summarize each conversation thread in the cluster
      2. Reflector (no-tag mode) per thread:
           Input: conversation summary
           Output: friction points, user corrections, what worked/didn't
      3. Semantic search against ~/.claude/skills/*
      4. LLM → new | update:X
      5. Curator (full skill builder mode):
           Input: all reflections + goal descriptions for this cluster
           Output: complete skill (YAML frontmatter, ## Workflow, ## Rules)
      6. Write skill to disk immediately
      7. Insert rule IDs into SQLite (counters = 0)
```

### Periodic Run

Reflector runs per-thread (tags rules, extracts insights). Curator runs per goal cluster (may span multiple skills).

```
New sessions (not yet processed)
  → extract goals from each new session
  → for each thread:
      summarize the conversation
      └ extract which skills were invoked (from conversation metadata)
      └ Reflector (tag mode):
          Input: skill ## Rules sections + conversation summary
          Output: per-rule tags (followed/violated_bad/violated_ok/not_relevant)
                  + new insights
      └ update SQLite rule counters
      └ mark session processed

  → After accumulating N threads:
      re-cluster recent threads by their goals
      → for each cluster:
          Curator:
            Input: all reflections + insights from threads in this cluster
                   + all skills referenced by those threads
                   + SQLite rule stats (helpful/harmful counts)
            Output: ADD operations for new rules (assigned to appropriate skills)
            → can also propose new skills if cluster has no matching skill
```

### Skill Structure

```markdown
---
name: skill-name
description: When to use, what it does
---

## Workflow
...

## Rules
- [skillname-00001] Always verify repo context before editing files
- [skillname-00002] Prefer minimal diffs over large refactors
```

**Important**: `helpful=N harmful=N` counters are NOT in the skill markdown — they would skew the model's behavior. Counters live only in SQLite. The skill markdown only has rule ID + content.

### State Tracking (SQLite)

- `rules` table: `id`, `skill_name`, `content`, `helpful_count`, `harmful_count`, `created_at`, `updated_at`
- `processed_sessions` table: `session_id`, `processed_at`, `skill_name`

## Questions & Answers

### Q1: Initial run — how to process 100s of conversations with overlap?

**Answer**: Sequential cluster processing, largest first. Reflector in no-tag mode, curator builds full skills.

1. Extract all goals, cluster, sort by size descending
2. For each cluster (≥3 goals):
   - Summarize each conversation thread
   - **Reflector** (no-tag mode) per thread: extracts friction points, corrections, what worked/didn't (no rule tagging since rules don't exist yet)
   - **Semantic search** against `~/.claude/skills/*` (includes pre-existing + newly created skills)
   - **LLM** decides `new` or `update:X`
   - **Curator** (full builder mode): takes all reflections + goal descriptions → produces complete skill (frontmatter, workflow, rules with IDs, counters=0 in SQLite)
   - Write skill to disk immediately
3. Skip clusters below min size (e.g. <3 goals)

**Why overlap is handled**: If cluster 3 overlaps with cluster 1's newly created skill, the semantic search finds it, and the curator merges instead of creating a duplicate.

**Optimization**: Don't summarize all threads upfront. Extract goals (cheap), cluster, then only summarize threads for clusters we're actually synthesizing.

---

### Q2: Periodic/daily runs — how reflector and curator work on an ongoing basis?

**Answer**: Reflector per-thread (tag mode), curator per goal cluster (may span multiple skills).

**Skill matching**: Skills invoked during the session are extracted from conversation metadata/transcript — no semantic search needed.

**Reflector** (runs per-thread, per-skill invoked):

Input: skill's `## Rules` section + conversation summary
LLM tags each rule:
- `irrelevant` → rule doesn't apply to this conversation → no counter change
- `followed_helpful` → rule was followed, outcome was good → `helpful++`
- `followed_harmful` → rule was followed, but it caused harm → `harmful++`
- `not_followed` → rule was relevant but agent didn't follow it → `harmful++`

Plus: extract new insights not covered by existing rules.

**Curator** (runs per goal cluster, after N threads accumulated):

Threads get re-clustered by goals. Curator runs on each cluster:
- Input: all reflections + insights from threads in this cluster, all skills referenced, SQLite rule stats
- Output: ADD operations for new rules (assigned to appropriate skills)
- Curator can propose a new skill if the cluster has no matching skill
- Update relevant SKILL.md files and SQLite

**Curator frequency**: Reflector runs every thread (lightweight tagging). Curator runs when enough threads accumulate in a goal cluster — batch at end of daily run or after N threads.

**Transition reflector → curator**: Reflector outputs stored per-thread. Periodically, threads are re-clustered by goals. Each cluster's accumulated reflections fed to curator. Curator synthesizes across multiple threads, deduplicates insights, assigns rules to skills.

**Open question**:
- At what harmful/helpful ratio flag a rule for review? (e.g. harmful > helpful * 3 → `suspicious`)

---

### Q3: First-time rules population — existing skills have no rules, how to bootstrap them?

**Resolved**: No separate bootstrapping. If the curator decides to update an existing skill, it adds the `## Rules` section on its own. No need for a separate population step.

---

### Q4: Reflector mechanics — how it tags rules from conversations?

**Tags** (4 categories):
- `irrelevant` — rule doesn't apply to this conversation → no counter change
- `followed_helpful` — agent followed the rule, outcome was good → `helpful++`
- `followed_harmful` — agent followed the rule, but it caused harm or a bad outcome → `harmful++`
- `not_followed` — rule was relevant but agent didn't follow it → `harmful++` (rule was there but wasn't used when it should have been)
